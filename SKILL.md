# 五大部门通用准则

每次编码任务自动遵循以下五个部门的规范。

---

## 部门1: 安全审计部

**每次修改代码前**执行 OWASP Top 10 安全检查。

### 禁止
- 硬编码密钥、密码、token、API key → 一律环境变量
- 前端存储敏感 token → 用 httpOnly cookie
- 用户输入直接拼接到 SQL/Shell/OS 命令
- 日志中打印密码、token、卡号、身份证号
- 禁用 HTTPS 验证 (`NODE_TLS_REJECT_UNAUTHORIZED=0`)

### 必须
- 所有 API 接口有权限校验
- 用户输入做参数校验和转义
- 文件上传校验类型和大小
- 密码用 bcrypt/argon2 hash
- API 有 Rate Limiting

### 输出格式
每次修改项目前，输出简洁清单：

```
## OWASP Top 10 审计
🔴 高危: [文件:行号 — 问题 → 修复]
🟡 中危: [...]
🟢 无
修复建议: 1. ... 2. ...
```

只报告风险，不改代码。不超过15行。

## 部门2: 代码规范部

**写任何代码前**过三条铁律。

### 铁律1: 先思考
- 陈述假设。不确定就先问
- 有多种解法全部列出，不默默选一个
- 有更简单方案主动说
- 遇到不清楚就停，问用户

### 铁律2: 极简
- 没有多余功能 — 只做用户要求的
- 没有一次性抽象 — 三条相似代码好过 premature abstraction
- 没有未要求的灵活性和可配置性
- 没有不可能场景的错误处理
- 200 行能变 50 行就重写

自问：资深工程师会觉得过度复杂吗？

### 铁律3: 精准
- 不改相邻代码的注释、格式
- 不重构没坏的东西
- 匹配现有代码风格
- 发现无关死代码，提一句但别删
- 只删自己改动造成的未使用 import/变量/函数

检验标准：diff 每行改动都能追溯到用户需求。

## 部门3: 分身协调部

**非平凡任务**自动拆分、并行执行。

### Agent 匹配表
| 任务 | Agent |
|------|-------|
| 前端 UI/React/Vue/CSS | Frontend Developer |
| 后端 API/数据库/架构 | Backend Architect |
| 安全审计/加密/鉴权 | Security Engineer |
| CI/CD/服务器/部署 | DevOps Automator |
| 通用代码 | general-purpose |

### 执行流程
1. 识别任务涉及的领域
2. 拆分为互不依赖的独立子任务
3. 并行启动多分身 (`run_in_background: true`)
4. 合并结果，向用户报告

### 原则
- 一个角色一个 Agent，不混用
- 默认并行，除非有强依赖
- 不自己大包大揽 — 用 Agent 分担

## 部门4: 执行验证部

**每批代码改动后**进入验证循环。

### 验证流程
1. **定义成功标准** — "加校验" → "写测试覆盖非法输入，测试通过"
2. **构建** → `npm run build`，失败 → 修复 → 重试
3. **部署**（如适用）→ `npx vercel --prod --yes`
4. **测试** → `npm test`，全绿才通过
5. **总结** → 报告做了什么、还有什么没做

### 输出格式
```
## 验证结果
- 构建: ✅/❌
- 部署: ✅/❌
- 测试: ✅/❌ (x/y)
## 进度
- ✅ 已完成: ...
- ⏳ 待完成: ...
```

### 原则
- 弱成功标准导致反复追问 → 花 30 秒定义清楚
- 构建+部署+测试缺一不可
- 验证失败诊断根因，不绕过

## 部门5: 内容质量部

**创建教学/文档/技术内容时**遵循标准。

### 每个知识点必须包含
| 要素 | 说明 |
|------|------|
| 生活类比 | 日常事物解释概念，不用学术术语 |
| 代码示例 | 先展示完整可运行代码，再解释 |
| 逐行拆解 | 每行代码作用 + 为什么这么写 |
| 常见误区 | 新手易踩坑 + 为什么错 + 怎么避 |
| 企业场景 | 真实项目怎么用 + 为什么选这个方案 |

### 写作原则
- 代码优先 — 先代码后解释
- 从0到企业级 — 零基础到项目运用
- 简洁易懂 — 术语要解释
- 不做表面功夫 — 深入讲「为什么」和「怎么用」

### 自检
写完自问：完全不懂的人能看懂吗？每行代码都有解释吗？有类比/误区/企业场景吗？
